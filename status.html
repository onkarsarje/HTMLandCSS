what are HTTP Status Codes and what do they mean.

When we search on google for a website neog.camp for instance the browser sends requests to the server and documents, style-sheets, javascript files, fonts are downloaded. And for each request there are multiple HTTP Headers which according to the MDN Web Docs let the client and the server pass additional information with an HTTP request or response. Here I will explain different Status Codes that will appear when a request is made to the server.

We can see what requests are sent, by checking the Inspect Network Tab which is opened by right-clicking mouse button and seclect "Inspect", and then opening "Network" tab as you can see in the image below. Now when the name of the website is entered and we press the search button, multiple requests are registered and we can see them on the display, along with the name of the request Status Codes can be seen.

The Status-Line part of the HTTP response message is used to inform the clients of their request's result. The standard status codes are divided into five categories and are used to convey the results to the clients' requests.

  • 1xx: Informational -> Communicates transfer protocol-level information.
  • 2xx: Success -> Indicates that the client's request was accepted successfully.
  • 3xx: Redirection –> Indicates that the client must take additional action to complete their request.
  • 4xx: Client Error –> This category of error status codes points the finger at clients.
  • 5xx: Server Error –> The server takes responsibility for these error status codes.

Here is a brief description of some of the Status Codes we encountered:

 ◼ 200: It indicates that the REST API successfully carried out whatever action the client requested. The information returned with the response is dependent on the method used in the request, for example: GET, HEAD, POST, TRACE.
 ◼ 204: It is a response to clients' PUT, POST, DELETE requests and is sent when the REST API declines to send back anu status message.
 ◼ 304: It is used when the resource has not been modified since the last version. Since the client already has a downloaded copy the resource need not be retransmitted.
 ◼ 401: It is sent when the client tried to operate on a resource without proper authorization. The client may have provided the wrong credentials or none at all for the protected resource.
 ◼ 404: It is sent when the REST API is unable to map the client's URL to a resource but may be available in future. If an old resource is permanently unavailable then 410(Gone) code should be used.
